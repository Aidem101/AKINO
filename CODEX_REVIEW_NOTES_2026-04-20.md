## 1. Критичные проблемы безопасности и целостности данных

### 1.1. Небезопасный OTP-вход
- При запросе кода сервер сам возвращает одноразовый код в JSON-ответе как `demoCode`.
- Файлы:
  - `public/api/request-code.php`
  - `src/auth.php`
- Точки:
  - `public/api/request-code.php`: в ответе явно отдаётся `'demoCode' => $request['code']`
  - `src/auth.php`: код создаётся в `create_auth_code_request()`
- Почему это плохо:
  - Любой клиент, получивший ответ API, уже знает OTP.
  - Это превращает "подтверждение по коду" в формальность.
  - Сейчас это допустимо только как dev/demo-режим, но в коде нет явного флага окружения, который бы это ограничивал.
- Что исправлять:
  - Не возвращать OTP в продовом ответе вообще.
  - Если нужен локальный demo-режим, завести явный конфиг-флаг вроде `AKINO_DEMO_AUTH=1`.
  - В UI показывать demo-код только при включённом dev-режиме.

### 1.2. Нет rate limit / защиты от перебора OTP
- Файлы:
  - `public/api/request-code.php`
  - `public/api/verify-code.php`
  - `src/auth.php`
  - `database/akino.sql`
- Проблема:
  - Нет ограничения на количество запросов OTP для номера.
  - Нет ограничения на количество попыток ввода кода.
  - В таблице `auth_codes` нет полей для счётчика неудачных попыток, lockout, IP rate limiting.
- Почему это плохо:
  - Код из 4 цифр слишком короткий без throttling.
  - Даже если убрать `demoCode`, вход остаётся уязвимым к перебору.
- Что исправлять:
  - Добавить лимит запросов кода на номер/IP.
  - Добавить лимит проверок кода на `request_id`.
  - После N неудачных попыток инвалидировать код.
  - Логировать частые запросы и частые ошибки подтверждения.

### 1.3. Подтверждение OTP неатомарно
- Файл:
  - `src/auth.php`
- Функция:
  - `verify_auth_code_request()`
- Проблема:
  - Сначала код читается `SELECT * FROM auth_codes WHERE id = :id`,
  - потом проверяется,
  - потом отдельным запросом ставится `used_at = NOW()`.
- Риск:
  - Два параллельных запроса могут успеть принять один и тот же код.
- Что исправлять:
  - Делать проверку и пометку использования атомарно.
  - Варианты:
    - транзакция + `SELECT ... FOR UPDATE`
    - либо `UPDATE ... WHERE id = :id AND used_at IS NULL AND expires_at >= NOW()` с последующей проверкой количества изменённых строк.

### 1.4. Session fixation для обычного пользователя
- Файл:
  - `src/auth.php`
- Функция:
  - `login_user()`
- Проблема:
  - После входа в аккаунт не вызывается `session_regenerate_id(true)`.
  - В админке это сделано правильно, в `admin_login_attempt()`.
- Риск:
  - Сессия пользователя не ротируется после логина.
- Что исправлять:
  - Добавить `session_regenerate_id(true)` в `login_user()`.
  - Проверить, что logout и повторный login корректно работают после этого.

### 1.5. Захардкоженные секреты в репозитории
- Файлы:
  - `config/admin.php`
  - `config/database.php`
  - `database/akino.sql`
  - `tests/smoke.php`
- Проблема:
  - В проекте лежат реальные дефолтные логин/пароль для админки.
  - В проекте лежит пароль пользователя БД.
  - Smoke-тесты завязаны на эти значения.
- Почему это плохо:
  - Любой, кто видит кодовую базу, уже знает учётные данные.
  - Это мешает безопасно деплоить проект даже на тестовый сервер.
- Что исправлять:
  - Вынести секреты в env / локальные конфиги вне git.
  - Убрать реальные значения из репозитория.
  - Обновить smoke-тесты так, чтобы они читали секреты из окружения или использовали временные тестовые креды.

### 1.6. Автосоздание дефолтного администратора
- Файлы:
  - `src/admin.php`
  - `config/admin.php`
- Функции:
  - `ensure_admin_support()`
  - `ensure_admin_default_account()`
- Проблема:
  - Если таблица `admin_accounts` пуста, проект сам создаёт администратора с логином/паролем из конфига.
- Почему это плохо:
  - На чистой БД сразу появляется предсказуемый админский доступ.
  - Это удобно для демо, но опасно как поведение по умолчанию.
- Что исправлять:
  - Убрать автосоздание из обычного request-path.
  - Сделать отдельную bootstrap-команду/скрипт инициализации.
  - Либо разрешать автосоздание только в локальном dev-режиме.

## 2. Архитектурные проблемы БД и инициализации

### 2.1. Миграции выполняются на обычных веб-запросах
- Файл:
  - `src/admin.php`
- Функция:
  - `ensure_admin_support()`
- Проблема:
  - Прямо во время загрузки страницы выполняются `ALTER TABLE` и `CREATE TABLE IF NOT EXISTS`.
- Риск:
  - Схема меняется в runtime.
  - Ошибки миграции превращаются в "админка недоступна", без нормальной диагностики.
  - Возможны частично применённые изменения.
- Что исправлять:
  - Вынести миграции в отдельный SQL/CLI-шаг.
  - На веб-запросах только проверять готовность схемы, но не менять её.

### 2.2. Seed-фильмы перезаписываются при обычной работе приложения
- Файл:
  - `src/movies.php`
- Функция:
  - `ensure_movie_library()`
- Проблема:
  - При каждом bootstrap вызывается вставка seed-данных через `INSERT ... ON DUPLICATE KEY UPDATE`.
  - Для уже существующих карточек seed-значения снова накатываются поверх текущих данных.
- Почему это опасно:
  - Админ может отредактировать встроенную карточку фильма, а затем проект на следующем запросе откатит часть полей назад.
  - Если админ меняет название, а логика slug тоже меняется, возможны дубликаты контента вместо обновления исходной карточки.
- Что исправлять:
  - Полностью убрать seed/upsert из runtime.
  - Seed должен запускаться один раз отдельно.
  - Если seed всё же нужен, делать его строго idempotent по отдельной команде, а не в `bootstrap`.

### 2.3. Seed сезонов и эпизодов тоже идут через runtime-upsert
- Файл:
  - `src/playback.php`
- Функция:
  - `ensure_playback_library()`
- Проблема:
  - Сезоны и эпизоды серий тоже вставляются/обновляются при обычном запросе.
- Риск:
  - Правки эпизодов через админку будут неустойчивыми.
  - Любая ручная корректировка seed-сериала может откатываться.
- Что исправлять:
  - Так же, как и для фильмов: вынести в отдельный seed/migration шаг.

### 2.4. Подписки не защищены от гонок
- Файлы:
  - `src/subscription.php`
  - `database/akino.sql`
  - `public/api/subscription.php`
- Проблема:
  - `get_active_subscription_row()` вызывается до начала транзакции.
  - Нет блокировки строки через `FOR UPDATE`.
  - В таблице `user_subscriptions` нет ограничения, запрещающего несколько строк для одного пользователя.
- Риск:
  - Два параллельных запроса могут:
    - потерять одно продление,
    - создать дубли подписок,
    - дать неверную конечную дату.
- Что исправлять:
  - Либо один ряд подписки на пользователя + `UNIQUE(user_id)`,
  - либо явная модель истории подписок, но с корректной блокировкой активной записи.

## 3. Проблемы API и error handling

### 3.1. Сырые внутренние сообщения исключений уходят в клиент
- Файлы:
  - `public/api/favorites.php`
  - `public/api/profile.php`
  - `public/api/progress.php`
  - `public/api/request-code.php`
  - `public/api/session.php`
  - `public/api/subscription.php`
  - `public/api/verify-code.php`
- Проблема:
  - Во многих `catch` клиенту отдаётся `... . $exception->getMessage()`.
- Почему это плохо:
  - Пользовательский фронт получает детали внутренних ошибок БД/схемы/SQL/конфига.
  - Это усложняет UX и раскрывает внутренности приложения.
- Что исправлять:
  - Наружу отдавать только безопасные пользовательские сообщения.
  - Подробности писать в лог.
  - Разделить `user-facing error` и `internal debug error`.

### 3.2. Нет явной CSRF-защиты на пользовательских POST API
- Файлы:
  - `public/api/favorites.php`
  - `public/api/profile.php`
  - `public/api/progress.php`
  - `public/api/subscription.php`
  - `public/api/request-code.php`
  - `public/api/verify-code.php`
- Проблема:
  - В пользовательских API нет CSRF-токенов.
  - Также нет проверки `Origin`/`Referer`.
- Примечание:
  - Сейчас есть `SameSite=Lax` у cookie, но этого недостаточно считать полноценной серверной защитой.
- Что исправлять:
  - Ввести CSRF-токен для state-changing запросов.
  - Или как минимум валидировать `Origin` / `Referer` для same-origin POST.

### 3.3. Логин API выдаёт лишнюю информацию о состоянии аккаунта
- Файл:
  - `public/api/request-code.php`
- Проблема:
  - По ответам можно определить:
    - существует ли пользователь,
    - заблокирован ли пользователь,
    - новый ли будет создан аккаунт.
- Почему это плохо:
  - Это облегчает enumeration по номеру телефона.
- Что исправлять:
  - Привести ответы к более нейтральной форме.
  - Не различать существующий/новый аккаунт в публичном ответе.

## 5. Админка: логика, поведение и рабочие риски

### 5.1. Админка зависит от runtime-создания схемы и default admin
- Это уже описано выше, но важно ещё раз:
  - админка не просто использует БД,
  - она сама создаёт необходимые таблицы и дефолтного админа при обычном запросе.

- Файлы:
  - `src/admin.php`
  - `src/movies.php`

  - Админка умеет редактировать фильмы/сериалы,

  - Админ будет видеть "сохранение прошло успешно", но данные позже могут "самооткатиться".

### 5.3. Надо отдельно проверить, можно ли сломать админский рабочий поток удалением seed-контента
- Текущее замечание:

  - Удаляется ли seed-фильм навсегда или воскресает.
  - Что происходит с сезонами/эпизодами при удалении seed-сериала.

### 5.4. Нужно проверить поведение при смене title/slug встроенных карточек
- Файл:
  - `src/admin.php`

  - `save_admin_movie()`

### 6.5. `cabinet.js` хрупко привязан к порядку DOM-полей профиля
