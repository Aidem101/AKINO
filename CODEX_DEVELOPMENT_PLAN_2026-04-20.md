# CODEX Development Plan

Дата: 2026-04-20
Проект: AKINO (`C:\OSPanel\home\project`)
Назначение: рабочий план развития проекта по этапам. Это не список багов и не общий обзор, а последовательность действий: что делать, в каком порядке, зачем, от чего это зависит и что считать завершением этапа.

## 1. Главная цель

Довести AKINO от красивого учебного прототипа до устойчивого демонстрационного онлайн-кинотеатра, у которого:
- есть безопасная и предсказуемая техническая основа;
- есть полноценное ядро просмотра;
- есть сильный пользовательский опыт;
- есть удобное администрирование контента;
- есть база для дальнейшего роста без постоянного хаоса в коде и данных.

## 2. Основной принцип

Развивать проект нужно не "по вдохновению", а слоями.

Неправильный путь:
- добавлять новые страницы, анимации и функции поверх нестабильного фундамента.

Правильный путь:
1. Стабилизировать архитектуру и данные.
2. Довести ядро видеосервиса.
3. Усилить персональный пользовательский опыт.
4. Углубить контентную и продуктовую часть.
5. После этого заниматься масштабированием, SEO и полировкой.

## 3. Что нельзя откладывать

Перед любыми "красивыми" фичами обязательно нужно решить:
- runtime migrations;
- runtime seed/upsert;
- секреты в коде;
- auth/session/security;
- предсказуемую конфигурацию;
- безопасные API-ответы.

Пока это не исправлено, проект будет вести себя нестабильно:
- админка сможет "сохранять" контент, который потом откатится;
- логин и подписка будут уязвимыми;
- новые фичи будут только увеличивать технический долг.

## 4. Этапы развития

## Этап 1. Техническая стабилизация

### Цель

Убрать основные архитектурные риски и сделать проект предсказуемым.

### Что сделать

#### 1.1. Убрать runtime migrations и runtime schema changes
- Вынести `ALTER TABLE` и `CREATE TABLE` из request-path.
- Убрать изменение схемы из `src/admin.php`, `src/movies.php`, `src/playback.php`.
- Сделать отдельный путь инициализации схемы:
  - SQL-миграции;
  - либо CLI-скрипт;
  - либо последовательный bootstrap dev/setup.

#### 1.2. Убрать runtime seed/upsert из bootstrap
- Seed-фильмы и seed-эпизоды не должны перезаписываться при обычном открытии страниц.
- Сиды должны запускаться отдельно и явно.
- Нужно гарантировать:
  - что админские правки seed-контента не откатываются сами;
  - что удалённый seed-контент не "воскрешается" на следующем запросе.

#### 1.3. Вынести секреты и конфигурацию из кода
- Пароль БД убрать из `config/database.php`.
- Логин и пароль админа убрать из `config/admin.php`.
- Завести env-конфиг или локальный конфиг вне git.
- Разделить окружения:
  - local;
  - test;
  - demo;
  - future production.

#### 1.4. Исправить auth/session/security-основу
- Добавить `session_regenerate_id(true)` в пользовательский login.
- Пересобрать OTP flow:
  - без возврата `demoCode` по умолчанию;
  - с dev-флагом для локального режима;
  - с rate limit;
  - с лимитом попыток;
  - с атомарной проверкой кода.
- Закрыть session fixation.

#### 1.5. Исправить пользовательские API
- Добавить CSRF или хотя бы проверку `Origin/Referer`.
- Убрать сырые `exception->getMessage()` из ответов.
- Ввести единый формат ошибок API:
  - безопасное пользовательское сообщение;
  - внутренний лог отдельно.

### Что считать завершением этапа

- Ни одна страница не меняет схему БД при обычном открытии.
- Ни один обычный запрос не перезаписывает seed-контент.
- Все секреты вынесены из репозитория.
- Пользовательский login ротирует сессию.
- OTP flow работает безопаснее и предсказуемее.
- Пользовательские API больше не светят внутренние ошибки наружу.

### Приоритет

Максимальный. Это нужно делать первым.

## Этап 2. Ядро видеосервиса

### Цель

Довести главное ядро сервиса: просмотр, прогресс, сериалы, возврат к просмотру.

### Что сделать

#### 2.1. Усилить страницу просмотра
- Нормализовать UX плеера.
- Улучшить экран заблокированного просмотра.
- Добавить более понятный progress/save state.
- Продумать сценарии:
  - просмотр фильма;
  - просмотр серии;
  - окончание серии;
  - переход к следующей серии.

#### 2.2. Довести модель сериалов
- Чётко разделить:
  - сериал;
  - сезон;
  - серия.
- Добавить полноценное отображение списка эпизодов.
- Показывать:
  - длительность;
  - просмотрено/не просмотрено;
  - текущую серию;
  - следующую серию.

#### 2.3. Исправить историю и прогресс
- Убрать запись истории из `Film_Page.php`.
- Записывать историю на реальный playback.
- Синхронизировать:
  - progress;
  - continue watching;
  - history.
- Исключить ложные просмотры.

#### 2.4. Усилить блок "Продолжить просмотр"
- Сделать его заметным на главной.
- Сделать его понятным в кабинете.
- Для сериалов показывать:
  - сезон/серию;
  - прогресс;
  - кнопку продолжения.

### Что считать завершением этапа

- Пользователь действительно может остановиться и вернуться к просмотру.
- Сериалы ощущаются как сериалы, а не как фильмы с эпизодами на одной странице.
- История и continue watching заполняются логично.

### Приоритет

Очень высокий. Это главное продуктовое ядро.

## Этап 3. Пользовательский опыт и персонализация

### Цель

Сделать сервис не просто каталогом, а персональным продуктом.

### Что сделать

#### 3.1. Укрепить кабинет
- Превратить кабинет из "страницы профиля" в "центр персонального опыта".
- Добавить более сильные разделы:
  - продолжить просмотр;
  - избранное;
  - просмотренное;
  - позже посмотреть;
  - уведомления.

#### 3.2. Улучшить избранное и пользовательские списки
- Удаление из избранного без лишних переходов.
- Более плотные карточки внутри кабинета.
- Возможность быстро сортировать пользовательские списки.

#### 3.3. Улучшить рекомендации
- Рекомендации по жанрам.
- Рекомендации по истории.
- Рекомендации по избранному.
- Исключение уже досмотренного там, где это мешает.
- Разделение:
  - персональное;
  - редакторское;
  - новое;
  - популярное.

#### 3.4. Улучшить уведомления
- Центр уведомлений в кабинете.
- Уведомления:
  - новая серия;
  - заканчивается подписка;
  - премьера в избранном;
  - возвращение к незавершённому просмотру.

### Что считать завершением этапа

- Пользователь получает ощущение "сервис меня знает".
- Кабинет становится полезным, а не формальным.
- Рекомендации и уведомления перестают быть декоративными.

### Приоритет

Высокий, но после стабилизации и после ядра просмотра.

Есть вещи, которые можно сделать относительно быстро, если нужен заметный результат без полной перестройки.

### Быстрые продуктовые улучшения
- Починить мобильный поиск.
- Довести continue watching на главной и в кабинете.
- Сделать более удобное избранное.
- Улучшить пустые состояния.
- Убрать старые публичные `login.php` и `login_code.php`.

### Быстрые технические улучшения
- Убрать сырые error messages из API.
- Добавить `session_regenerate_id(true)` в пользовательский login.
- Привести asset versions к единой системе.
- Убрать дублирующий inline JS и `console.log`.

## 6. Что делать нельзя раньше времени

### Не стоит делать до стабилизации фундамента
- сложную оплату;
- многоуровневые рекомендации;
- детские профили;
- сложную социальную механику;
- большой рефактор UI-компонентов;
- масштабную SEO-подготовку.

Причина:
- всё это будет строиться на нестабильной основе и только увеличит стоимость последующей переработки.

## 9. Финальный вывод

Проект уже достаточно хороший, чтобы его развивать не как "набор страниц", а как систему.

Но дальше всё упирается в дисциплину:
- сначала фундамент;
- потом ядро;
- потом персонализация;
- потом глубина контента;
- потом масштабирование.

Если идти в этом порядке, AKINO можно превратить в очень сильный демонстрационный онлайн-кинотеатр.

Если идти в другом порядке, проект будет визуально становиться богаче, но технически тяжелее, хрупче и дороже в поддержке.
